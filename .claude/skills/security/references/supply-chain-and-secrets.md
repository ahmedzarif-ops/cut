# Dependency + Secret + Supply-Chain Audit — security reference
> Defensive reference for Cyrus (security). Condensed from a public defensive-security skills library (Apache-2.0), grounded in a typical modern web stack. DEFENSIVE ONLY: checklists, detections, and hardening to VERIFY on systems you own. Never run offensive/exploit steps; for any 'exploiting-*'/'testing-*' source, extract only the defensive control + how to confirm it, not an attack recipe.

## What Cyrus checks here
Whether the software supply chain is trustworthy end-to-end: that no secret has leaked into git, that every dependency is the genuine (non-malicious, non-typosquat) package at a non-vulnerable version, and that there is an automated gate (CI) catching all of this before it reaches the live deploy. Surfaces touched: the repo (`package.json`, `package-lock.json`, git history, `.env*` handling), the CI/CD pipeline (often absent early), and any separate repo (e.g. a multi-tenant product) with its own lockfile + MCP servers. This playbook typically **owns the "stand up CI + automated dependency updates" backlog item** and overlaps the "upgrade the framework to clear the dependency audit" item.

## Checklist

1. **Is there a CI pipeline at all?** If there is no `.github/` (or equivalent): no Actions workflow, no dependency audit, no typecheck/lint/build gate, no automated dependency updates — every commit reaches the host with zero automated security checks. **Where:** repo root, `package.json` scripts. **Severity: High** (usually the "do this first" backlog item). **Fix:** add a CI workflow running `npm ci` → tests → `tsc --noEmit` → lint → build → `npm audit --audit-level=high`, plus automated dependency updates (Dependabot/Renovate) for the `npm` and `github-actions` ecosystems (weekly). Fail the build on high/critical audit findings.

2. **No automated dependency vulnerability scan; framework advisories unpatched.** Without a dependency audit in CI (item 1), known CVEs in the tree go unnoticed; an outdated framework often carries advisories (e.g. a DoS in an image/optimization path). **Where:** `package.json`, `package-lock.json`. **Severity: High.** **Fix:** run SCA every build — `npm audit`, or Grype/Snyk/`osv-scanner --lockfile=package-lock.json` (OSV also flags `MAL-` malicious-package advisories). Track the framework upgrade to clear residual advisories. For unfixable-but-unreachable findings, document a time-boxed, expiring ignore (Snyk `.snyk` / Grype `.grype.yaml`) — never a silent permanent suppression.

3. **No secret scanning (gitleaks/TruffleHog) on commits or history.** Nothing blocks a future accidental commit of the cookie-signing secret, the payment secret + webhook secret, the cron secret, the database service/admin key, or LLM/telephony/email tokens. **Where:** repo-wide; secrets are referenced from your server/config modules. **Severity: High.** **Fix:** add a gitleaks pre-commit hook (`gitleaks protect --staged --redact`) AND a CI job (`gitleaks detect` + TruffleHog `--only-verified` on PRs, `fetch-depth: 0`); upload SARIF. Seed a `.gitleaks.toml` with custom rules for YOUR secret names + provider key formats (e.g. `sk_live_`/`whsec_`, database JWTs). Establish a baseline once so it gates only NEW secrets.

4. **Confirm env hygiene is actually clean.** Verify `.gitignore` covers `.env`, `.env.local`, `.env*.local`; that no real `.env*` file is git-tracked; that only `.env.example` (placeholder-only) is committed; and that any tracked config module holds **no** real secret (dev-only fallback + throws in prod). **Where:** `.gitignore`, `.env.example`, your cookie-signing/config module. **Severity: Low (verify-only).** **Fix:** keep `.env.example` placeholder-only; never commit a real `.env`; if item 3's history scan ever surfaces a real key, rotate at the provider first, then scrub history with `git filter-repo`/BFG — deleting the commit alone does not remove it from forks/caches.

5. **Map exposed-credential blast radius to financial fraud (F3).** A leaked payment secret or email/telephony token is not just data loss — it enables account takeover and monetization (refund fraud, spam-on-your-dime). The TruffleHog skill maps this to F3 `F1006.001 Exposed API Key`. **Where:** all provider tokens in env; the payment secret is the highest-value target. **Severity: High (if any leak found).** **Fix:** treat any verified-active key finding as an incident: deactivate/rotate immediately, review the provider's audit log for unauthorized use, then scrub.

6. **No malicious-package / install-script gate before install.** The host runs the install on every build; npm auto-runs `preinstall`/`install`/`postinstall` scripts with full privileges before app code — the exact vector of the Shai-Hulud worm and `event-stream`. **Where:** `package.json` deps, `package-lock.json`. **Severity: Med.** **Fix:** in CI, run `osv-scanner --lockfile=package-lock.json` (flags `MAL-` advisories) and optionally GuardDog `npm verify package.json`; review any dependency declaring lifecycle scripts. Pin builds to the lockfile via `npm ci` (not `npm install`) so resolved hashes can't drift; consider `--ignore-scripts` where the toolchain allows.

7. **Verify dependencies are the genuine packages, not typosquats (on every add).** A small runtime dep set of canonical packages is low risk today, but an agentic org may add packages autonomously. **Where:** any diff to `package.json`/`package-lock.json`. **Severity: Low-Med.** **Fix:** on each new dep, sanity-check name against the popular target (Levenshtein 1-2 from a known package = suspect), publish date, weekly downloads, and maintainer; reject low-download near-namesakes. This is a code-review gate, not a takedown tool.

8. **Dependency-confusion risk is low but must be re-verified if internal scopes appear.** With no `.npmrc` and only public scopes, there is no private package shadowable from the public registry. Risk rises if you ever publish an internal-named package. **Where:** `package.json` scopes; `.npmrc` (absent). **Severity: Low.** **Fix:** if an internal scope is introduced, add a committed `.npmrc` pinning that scope to the private registry (single virtual feed, never `extra-index-url` which picks highest version), and defensively register the public name as an empty placeholder.

9. **Lockfile integrity is unenforced in builds.** No CI means nobody verifies the deploy install matches `package-lock.json`. **Where:** host build settings, `package-lock.json`. **Severity: Med.** **Fix:** set the install command to `npm ci` (errors on lockfile/manifest mismatch and installs exact pinned versions). Commit the lockfile and review lockfile diffs in PRs for unexpected resolved-URL or integrity-hash changes (a swapped registry host is a red flag).

10. **Generate an SBOM for visibility + continuous re-scan.** You can't patch what you can't see; a stored SBOM lets new CVEs be matched against unchanged artifacts without rebuilding. **Where:** repo + any sibling repo. **Severity: Med.** **Fix:** `syft dir:. -o cyclonedx-json` then `grype sbom:<file> --fail-on high`; store the SBOM and re-scan on Grype DB updates. Satisfies the inventory half of supply-chain defense (NIST CSF `ID.AM-08`).

11. **When CI lands (item 1), harden the workflow itself — don't trade one risk for another.** A CI pipeline becomes a high-value target (it can hold deploy creds). **Where:** future `.github/workflows/*`. **Severity: Med (forward-looking).** **Fix:** pin every `uses:` to a full commit SHA (not `@v4` mutable tags) and let dependency updates bump them; set `permissions: {}` at top, grant least-privilege per job; never interpolate untrusted input (`github.event.pull_request.title`, branch names) directly into `run:` — pass via `env:`; avoid `pull_request_target` with PR-code checkout; protect `.github/` with CODEOWNERS; prefer OIDC over long-lived stored secrets.

12. **Bring every separate repo into the same gate.** If you run a second app (e.g. a multi-tenant product) with its own database, its own `package.json`/lockfile, and multiple MCP servers, it is a larger dependency + token surface than the main app. **Where:** the sibling repo. **Severity: Med.** **Fix:** apply items 1-11 there too; MCP server packages and their tokens are supply-chain + secret surface, so secret-scan and SCA-scan that repo independently.

13. **Third-party agent-tool adoption is an instruction-injection surface, not just a dependency.** When adopting any third-party agent tool / "reach" package / skill pack, check TWO things beyond the usual SCA: (a) whether the installer WRITES INTO the agent trust boundary — `~/.claude/skills`, `~/.agents/skills`, or the repo's `.claude/` — because a skill dropped there is auto-loaded into agent context and executes with the agent's tools; and (b) whether any bundled `SKILL.md`/agent doc instructs agents to FETCH + FOLLOW a remote document at runtime, which turns an external, mutable URL into live instructions. **Treat both as an OWASP LLM01 (prompt/instruction-injection) surface AND a supply-chain risk**, not merely a package version to audit. **Where:** any third-party skill/agent/tool install; its install script's write targets; every bundled `SKILL.md` for `curl`/`fetch`-then-follow patterns. **Severity: High (if either is present).** **Fix:** never auto-run the installer; inspect exactly what it writes and where; reject any skill that fetches-and-follows a remote doc (pin/vendor the content locally instead); owner-gate the adoption. This is the discipline behind a safe research skill that wraps only fixed, pinned read tools and never installs a reach package or follows a remote doc.

## Out of scope / do NOT do
- Never **execute** the offensive halves: do not detonate/install a suspect package outside an isolated throwaway VM, do not register/claim public package namespaces (the dependency-confusion "claim" step), and do not run namespace-resolution attacks — Cyrus is read-only advisory.
- Cyrus **recommends and verifies**, it does not deploy: standing up CI, rotating a leaked key, force-pushing scrubbed history, or upgrading the framework are owner-gated changes (engineer), not actions Cyrus performs.
- Skip heavyweight enterprise machinery (Dependency-Track server, paid Snyk seats, signed Cosign attestation infra) unless the small-product footprint clearly warrants it — `npm audit` + gitleaks + `osv-scanner` in a free CI workflow clears the "stand up CI" item at near-zero cost.
