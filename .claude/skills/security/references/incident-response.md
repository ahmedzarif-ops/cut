# Incident-response runbook (proportionate, single-owner) — security reference

> Defensive reference for Cyrus (security). Condensed from a public defensive-security skills library (Apache-2.0), grounded in a typical modern web stack. DEFENSIVE ONLY: checklists, detections, and hardening to VERIFY on systems you own. Never run offensive/exploit steps; for any 'exploiting-*'/'testing-*' source, extract only the defensive control + how to confirm it, not an attack recipe.

## What Cyrus checks here
Whether the app has a *proportionate* incident-response capability for a small, live, single-owner product — not an enterprise SOC/SOAR program. Cyrus verifies there is a documented runbook with one Incident Commander (the owner, `company.yml` → `owner.name`), the right detection signals on the managed-cloud dashboards, a known credential-rotation order, restorable backups, and a blameless lessons-learned loop that feeds the security backlog. Surfaces touched: the host, the database, the payment processor, the telephony/email/LLM providers, the git host, the signed access-cookie system, and the inbound webhook/cron trust boundaries. The goal is fast, reversible containment by one person, not a 5-person on-call rotation.

## Checklist
1. **A proportionate IR runbook actually exists** — Confirm a one-page runbook (this file is the template) names the owner as sole Incident Commander, lists vendor emergency/support contacts (host, database, payment processor, telephony, LLM provider, email provider, git host), the credential-rotation order (item 4), and when to pull the legal/compliance gates (Lex, Gus). Do NOT build SOAR, on-call paging, or 10 separate playbooks — that's over-engineering for one owner. *Where:* repo `SECURITY.md` + this dir. *Severity:* Med. *Fix:* keep current; run a 30-min tabletop quarterly.
2. **Detection sources are wired (no SIEM here)** — Triage relies on managed dashboards, so confirm each is actually watched with alerts: host runtime logs/alerts, database logs + security advisors, payment-processor dashboard (fraud rules + failed-webhook alerts), and spend/usage alerts on the LLM/email/telephony providers. *Where:* vendor consoles; your API route logs. *Severity:* High. *Fix:* enable hard billing/spend alerts on every paid vendor (abuse = cost-DoS) — maps to the durable-rate-limiting backlog item.
3. **Severity matrix scaled to the product** — Verify the runbook classifies: **P1** = money or PII exposure (leaked payment secret/webhook secret, leaked database service key, customer PII read, or entitlement bypass); **P2** = single-account / vendor-account takeover; **P3** = cost/abuse spike on LLM/email/SMS; **P4** = noise. Note the fail-OPEN risk: if the access gates grant access on DB error, a database outage *plus* an access attempt is a P1, not a P3. *Severity:* High. *Fix:* close the loop with the fail-closed-gates backlog item.
4. **Credential-rotation order is documented — and payment-secret rotation won't log out buyers** — The core containment lever is rotating secrets in host env + redeploy. Critical caveat: if the cookie-signing key derives from a fallback chain (a dedicated signing secret → falls back to the payment webhook secret → then the payment secret), and the dedicated secret is unset, rotating a payment secret silently re-keys every access cookie and mass-logs-out all buyers mid-incident. *Where:* your cookie-signing module. *Severity:* High. *Fix:* verify a dedicated signing secret is set explicitly in host env; document that rotating *it* is the deliberate "log everyone out" switch (item 5), separate from payment-secret rotation.
5. **Mass session-invalidation kill-switch exists** — If there is no token expiry today, a stolen access/session cookie (MITRE F3 F1004 stolen session cookie) is valid forever. The only revocation is rotating the dedicated signing secret in host env + redeploy, which invalidates all cookies at once. *Where:* your cookie-signing + access modules. *Severity:* High. *Fix:* document + test this switch; prioritize the token-exp and access-code-off-URL backlog items to shrink the blast radius.
6. **Service/admin database-key compromise is the worst-case (ransomware-equivalent)** — The admin client bypasses RLS; if leaked, an attacker can read, delete, or hold all data hostage. Verify: (a) the key is server-only behind the runtime browser guard and never in the client bundle (`grep` build output for the key prefix), (b) point-in-time-recovery / daily backups are enabled and a restore was actually tested, (c) rotation is documented (rotate in the DB → update host env → redeploy). *Severity:* Critical. *Fix:* maps to the server-only-split backlog item.
7. **Backups are *restorable*, not just configured** — For data destruction/extortion, recovery is restore-not-pay. Confirm DB backups/PITR are on, the Git repo is the code-of-record, and a test restore has been run (3-2-1 adapted: DB backup + Git + an offsite export). *Severity:* High. *Fix:* schedule a real restore drill; never pay extortion (run OFAC/legal via Lex first if ever raised).
8. **Inbound trust boundaries survive an incident** — After any secret rotation, the webhook/cron verifiers must still pass: payment sig via signature construction, telephony via provider validation, cron via constant-time Bearer secret. *Severity:* Med. *Fix:* if the payment webhook secret or cron secret is rotated during IR, redeploy and re-run the `deploy-and-verify` smoke battery on these endpoints.
9. **Phishing the single owner = the top realistic intrusion** — The owner is the sole admin of the host, database, payment processor, git host, LLM provider, and email/telephony; one phished/AiTM credential = full org compromise (MITRE F3 F1006 account takeover, F1004 session-cookie theft). Verify phishing-resistant MFA (passkeys/hardware key) on ALL provider accounts, with no SMS-only fallback. *Where:* infra (no source file). *Severity:* Critical. *Fix:* containment for a confirmed owner phish = revoke sessions + rotate creds on every vendor in the item-4 order, then rotate the dedicated signing secret.
10. **Email-channel hygiene (spoofing in + out)** — Transactional mail goes via your ESP and any outbound via a separate sending domain (`outbound`). Verify SPF/DKIM/DMARC (target `p=reject`) on the sending domain so attackers can't spoof you to phish buyers, and that unsubscribe-over-GET isn't abusable for enumeration during a campaign. *Severity:* Med. *Fix:* maps to the unsubscribe-over-GET backlog item.
11. **SSRF guard intact during an incident** — User-supplied-URL fetches (site scan/rescan) go through the SSRF-guarded fetch (http/https + standard ports, DNS-resolve + reject private/reserved IPs, re-validate every redirect hop, byte cap). A regression here lets an attacker pivot to cloud metadata. *Severity:* Med. *Fix:* confirm the guard's redirect re-validation and private-IP rejection still hold after any change; MITRE ATT&CK T1190.
12. **Per-route cost/abuse kill-switch (LLM + email + SMS)** — Cost-based DoS is the most likely "attack," and if current limits are in-memory/per-instance (rate-limiting backlog item), verify a fast lever exists to cap or disable each paid route via a host env flag + redeploy and via hard vendor spend caps. *Where:* your paid routes. *Severity:* High. *Fix:* add a documented per-route disable flag + provider spend ceilings.
13. **Post-incident → durable fix, not just cleanup** — Verify the loop closes blamelessly: run 5-Whys RCA, compute dwell / MTTD / MTTC / MTTR, then convert each root cause into (a) a security backlog item (or promote an existing one) and (b) a regression test/drift guard per your build convention — logic in your scripts dir with a paired test, gated by the ship bar (tests green + config-validate clean + clean build). "Write a new detection rule" maps to "write a new automated test" here. *Severity:* Med.
14. **Breach-notification is gated, not improvised** — If PII exposure is confirmed (your database's contact/pipeline/phone data — not cards, which the payment processor carries as SAQ-A), the runbook routes to Lex (`legal`) and Gus (`compliance-review`) before any customer or regulator notice. *Severity:* High. *Fix:* never self-author a breach notice; the specialist gates own external statements.

## Framework tags
- **MITRE ATT&CK:** T1078 (Valid Accounts), T1566/.001/.002 (Phishing), T1204.001/.002 (User Execution), T1114 (Email Collection), T1056.003 (Web Portal Capture), T1190 (Exploit Public-Facing App), T1041 (Exfil over C2), T1486 (Data Encrypted for Impact), T1490 (Inhibit System Recovery), T1489 (Service Stop), T1070 (Indicator Removal), T1537/T1580/T1525 (cloud abuse, discovery, implant).
- **MITRE F3 (fraud):** F1006/.002 (Account Takeover / Exposed Login Credential), F1004 (Access with Stolen Session Cookie), F1020.002 (Fake Website), F1018 / F1017 / F1025.003 (monetization: crypto, physical instruments, wire transfer) — relevant only as extortion/cash-out context, never as actions.
- **NIST CSF 2.0:** RS.MA-01/-02 (Incident Management), RS.AN-03 (Analysis/forensics), RC.RP-01 (Recovery Plan Execution).
- **OWASP (relevant):** Top 10 A07 Auth Failures + A09 Logging/Monitoring (items 2, 5, 9); API Top 10 (webhook/cron auth, item 8); LLM Top 10 unbounded-consumption (item 12 spend abuse).
- *(Not carried by these sources: D3FEND, NIST AI RMF, ATLAS — out of scope for this playbook.)*

## Source-skill taxonomy (read deeper, never execute)
- `building-incident-response-playbook` — **primary.** Playbook structure, RACI, decision trees, severity criteria; mine for the proportionate one-owner template (items 1, 3).
- `triaging-security-incident-with-ir-playbook` — **primary.** Severity scoring + enrichment; adapt its SIEM/SOAR flow down to your dashboards + spend alerts (items 2, 3).
- `conducting-cloud-incident-response` — **primary.** Most relevant: identity-first containment, credential rotation, evidence from cloud logs, IMDS/SSRF hardening (items 4-8, 11).
- `performing-ransomware-response` — **primary (read defensively).** Backup integrity, 3-2-1-1-0, restore-not-pay, OFAC/legal gating; map to service-key / data-destruction worst case (items 6, 7).
- `conducting-phishing-incident-response` — **primary.** Owner-phish + AiTM session-cookie theft, MFA, account containment, SPF/DKIM/DMARC (items 9, 10).
- `conducting-post-incident-lessons-learned` — **primary.** Blameless RCA, dwell/MTTD/MTTC/MTTR metrics, turn findings into tests/backlog (item 13).

## Out of scope / do NOT do
- Do not run the offensive siblings (`exploiting-*`, `testing-*`, GoPhish-style simulation, malware detonation, decryptor execution) — Cyrus is read-only advisory; extract only the defensive control + how to confirm it.
- Do not build enterprise machinery a single-owner product won't staff: SOAR automation, 24/7 on-call paging, a dozen per-type playbooks, or live malware forensics. One Incident Commander + reversible cloud-console levers is the right altitude.
- Do not authorize ransom payment, breach notices, or law-enforcement contact from this runbook — those route to the owner + the Lex/Gus gates.
