# Payments, Rate-Limiting & Abuse — security reference
> Defensive reference for Cyrus (security). Condensed from a public defensive-security skills library (Apache-2.0), grounded in a typical modern web stack. DEFENSIVE ONLY: checklists, detections, and hardening to VERIFY on systems you own. Never run offensive/exploit steps; for any 'exploiting-*'/'testing-*' source, extract only the defensive control + how to confirm it, not an attack recipe.

## What Cyrus checks here
The money-and-resource abuse surface: paid LLM/email endpoints (denial-of-wallet), the checkout/webhook path, the entitlement gates that decide who may spend or read PII, any shared access-code bypass (the closest thing to a "login" to brute-force), and owner-tool object-access scoping (BOLA/IDOR + enumeration). It folds in the MITRE F3 fraud lens — how an intrusion turns into money — which for a product whose payment processor owns all real funds-movement means **resource theft** (free LLM/email spend on a stolen privileged cookie) and **paywall bypass**. Surfaces touched: your API routes, your access/entitlement module, your cron-auth module, and the rate-limiting / fail-closed / token-expiry backlog items.

## Checklist
1. **Rate limits are in-memory per warm instance, not durable.** Limiters that use a module-scoped `Map` break on serverless: each instance has its own Map, so a distributed caller bypasses the cap by spreading requests across instances — the "in-memory without shared state allows limit bypass" pitfall. **Severity: High.** Fix: move to a shared store (a KV/Redis/Upstash-style backend) with a sliding-window or token-bucket counter keyed per IP **and** per owner id; keep the in-memory Map only as a fast-path pre-filter.
2. **Denial-of-wallet on the LLM endpoints.** Any route that calls the LLM provider on every allowed request has its true ceiling in the provider's monthly spend cap. **Severity: High.** Verify a hard monthly spend cap actually exists in the LLM provider console AND an email-provider monthly send cap exists — these are the only true ceilings until durable rate-limiting lands. Map: OWASP LLM10:2025 Unbounded Consumption; ATT&CK T1190.
3. **Client-controlled `x-forwarded-for` as the rate-limit key.** If every limiter keys off `req.headers.get("x-forwarded-for").split(",")[0]`, a caller can forge the leftmost XFF to rotate "identities" and defeat the per-IP cap — the "trusting X-Forwarded-For without validating against the LB" pitfall. **Severity: Med.** Fix: use the platform-provided client IP (`request.ip` / the rightmost trusted hop), not the raw leftmost XFF.
4. **A shared access-code bypass = the brute-force / credential-stuffing target.** Any single shared access/comp code compared server-side and rate-limited in-memory only is guessable by a distributed attacker (many IPs/ASNs) who bypasses the per-IP, per-instance cap — the credential-stuffing indicator set (many IPs, low success rate, hosting/VPS ASN concentration). **Severity: High.** Fix: durable per-IP limit + a global failed-attempt counter with backoff + alert on bursts of wrong codes; move the code off any URL query string to a POST/login UI; keep the constant-time compare and fail-closed-when-unset behavior.
5. **A privileged session cookie is a replayable bearer token with no expiry — the pass-the-cookie risk.** A payload of only `{tier, id, iat}` — no `exp`, no device binding — means a leaked/stolen privileged cookie is replayable indefinitely from any IP to spend LLM/email budget and read owner PII. This is the local analog of SSO/OAuth token-replay (T1550.001). **Severity: High.** Fix: add `exp` + a refresh path; build a session revocation list; short token lifetime + rotation.
6. **Entitlement gates fail OPEN on DB error — the F3 monetization gap.** Gates that `catch {}` and return the payload mean that during a DB blip a **canceled/lapsed** subscriber (status row says `canceled` but the read failed) keeps spending LLM budget and reading PII — exactly "convert access into ongoing resource theft." **Severity: High.** Fix: make the money-spending (LLM) and PII gates fail **closed** or grace-window, and reconcile from the source of truth; leave low-risk content gates fail-open if desired.
7. **Owner object-access is id-scoped — verify it stays that way (BOLA/IDOR).** Confirm every owner route filters `.eq("<owner_key>", s)` so an owner can only touch their own rows. **Watch child-table reads** scoped only by a parent id with no owner filter — safe only because the parent was first loaded owner-scoped and returns 404 if not owned. **Severity: Med.** Fix: keep the ownership check before every child read; add a regression test asserting another owner's id returns 404, not data. Map: OWASP API1:2023 BOLA; ATT&CK T1087.
8. **Identifiers are unpredictable — confirm no sequential public IDs leak (enumeration).** Prefer UUIDs / signed tokens over sequential integer ids. **Severity: Low.** Verify no owner-facing route exposes a sequential id an attacker could iterate, and that list endpoints never return other owners' object ids.
9. **Payment webhook trust + idempotency.** Confirm the webhook verifies every event via signature construction (the signature is the only trust path) and dedupes one-time side effects (check for an existing payment row before granting/emailing). **Severity: Med.** Verify: (a) entitlement is derived only from the verified event, never from client-controllable metadata; (b) the dedupe is safe under at-least-once concurrent re-delivery (a race could double-grant/double-email) — prefer an idempotency key or unique constraint on the session id. Map: OWASP API6:2023.
10. **Price & coupon are server-computed — keep it that way.** Confirm the checkout computes amounts and any discount server-side and the client cannot set price/coupon. **Severity: Low.** Verify discount codes stay server-only and a code can't be replayed to discount a tier it shouldn't, or stacked.
11. **No CI / dependency audit / auto-updates + an outdated framework leave a DoS-class advisory open.** A framework's image/optimization DoS is a resource-exhaustion bug rate-limiting cares about. **Severity: Med.** Fix: stand up CI (tests + typecheck + lint + build + dependency audit) + auto-updates, then upgrade the framework to clear it.
12. **Abuse observability is thin — wire detection signals to the data machine.** Adaptive rate-limiters escalate on tracked auth-failures, error-rate, and unique-endpoints per client; token-abuse hunts alert when one session is seen from multiple IPs/ASNs. Log none of this and you're blind. **Severity: Med.** Fix: log 429s, 403s, wrong-access-code attempts, and any single owner id used from many IPs/ASNs (token-replay tell) to the measurement layer (Dana / `data-analytics`) with alert thresholds. Bonus: return standard `Retry-After` + `X-RateLimit-*` headers on 429s.

## Framework tags
- **MITRE ATT&CK:** T1190 (Exploit Public-Facing App / DoS via paid endpoints), T1110 + T1110.003 (Password Spraying) + T1110.004 (Credential Stuffing — access-code guessing), T1078 / T1078.004 (Valid Accounts: Cloud — replayed cookie), T1550.001 (Use Alternate Auth Material: App Access Token — privileged-cookie replay), T1539 (Steal Web Session Cookie), T1528 (Steal App Access Token), T1595 / T1595.002 (Active Scanning), T1087 (Account/Object Discovery), T1046, T1552.001, T1098.001.
- **MITRE F3 v1.1 (fraud lens):** tactics **Positioning (FA0001)** and **Monetization (FA0002)**, plus Initial-Access (TA0001). Techniques: **F1006 Account Takeover** + **F1006.002 Account Takeover: Exposed Login Credential** (leaked access code / stolen cookie), F1005.003 Account Manipulation: Add Beneficiary (analog: tampering with owner data post-access). NOTE: the F3 cash-out techniques (F1025.003 Wire Transfer, F1018 Convert to Cryptocurrency) are the **payment processor's** scope — if you have no funds-movement surface, monetization here = resource/LLM-spend theft, not EFT.
- **NIST CSF 2.0:** PR.PS-01, ID.RA-01, PR.DS-10, DE.CM-01, DE.AE-02, RS.MA-01, GV.OV-01 (carried by the source skills).
- **OWASP:** API1:2023 BOLA, API2:2023 Broken Authentication, API4:2023 Unrestricted Resource Consumption, API6:2023 Unrestricted Access to Sensitive Business Flows; Web A01 Broken Access Control, A07 Auth Failures; **LLM10:2025 Unbounded Consumption** (denial-of-wallet on the LLM endpoints); Agentic — tool/resource abuse (relevant to any MCP fleet).
- **D3FEND (defensive):** Resource Access Pattern Analysis, User Behavior Analysis, Account Locking, Credential/Session Rotation. **NIST AI RMF:** MEASURE/MANAGE for LLM spend governance.

## Source-skill taxonomy (read deeper, never execute)
- `implementing-api-rate-limiting-and-throttling` — primary. Per-user/per-IP/per-endpoint limits, stricter auth-endpoint limits, tiered quotas, the in-memory-bypass and XFF-trust pitfalls, 429 + Retry-After hygiene. Translate the Redis/Kong/Flask code to your serverless KV.
- `implementing-api-abuse-detection-with-rate-limiting` — primary. Token-bucket/sliding-window/adaptive algorithms, threat-scoring on auth-failures/error-rate/unique-endpoints, escalating block durations. Mine the threat-assessment signals for item 12.
- `hunting-credential-stuffing-attacks` — primary (carries F3). Indicators for the access-code brute-force hunt: many IPs per target, <1% success, hosting/VPS ASN concentration, UA uniformity, password-spray shape.
- `hunting-saas-sso-token-abuse` — primary. Pass-the-cookie / token-replay detection (one session from multiple IPs/ASNs, impossible travel, refresh reuse) + preventive controls (short token lifetime, rotation, binding). Translate Entra/Okta concepts to your privileged cookie even if you have no SSO.
- `detecting-api-enumeration-attacks` — primary. BOLA/IDOR + sequential/UUID enumeration detection and the server-side authz + unpredictable-id prevention controls behind items 7-8.

## Out of scope / do NOT do
- Never run the library's offensive siblings (any `exploiting-*` / `testing-*` enumeration, brute-force, or token-theft tooling) against your app, customers, or third parties — extract the defensive control only.
- Do not deploy the skills' reference infra (Redis Cluster, Kong/NGINX/Envoy, Okta/Entra) if the app is serverless + managed-DB with no SSO; translate the concepts (durable counter, token expiry, session-replay detection), don't lift the stack.
- Funds-movement, chargeback/refund cash-out, and PCI/card handling are the payment processor's scope (SAQ-A) — Cyrus verifies you never trust client price/coupon and always verify the webhook signature, but does not re-implement payment-fraud controls the processor already owns.
