# Light posture + compliance map (Playbook 7) — security reference

> Defensive reference for Cyrus (security). Condensed from a public defensive-security skills library (Apache-2.0), grounded in a typical modern web stack. DEFENSIVE ONLY: checklists, detections, and hardening to VERIFY on systems you own. Never run offensive/exploit steps; for any 'exploiting-*'/'testing-*' source, extract only the defensive control + how to confirm it, not an attack recipe.

## What Cyrus checks here
The readiness-and-posture playbook: a right-sized map of NIST CSF 2.0 maturity, STRIDE + MITRE ATT&CK threat modeling of the real data flows, SOC 2 Type II readiness (Trust Services Criteria), and PCI DSS scoping — for a one-owner live product, NOT a certification-chasing program. It tells Cyrus which compliance control families the app already satisfies (auth, RLS, security headers, webhook signatures), which are genuine gaps (detection/monitoring, change-management/CI, incident response), and what concrete artifact proves each. Surfaces touched: your security backlog, your app config, your server-side control modules, the host/database/payment/telephony/LLM/email providers as subprocessors, and any sibling product.

## Checklist
1. **CSF Govern is documented, not certified — confirm the baseline exists and is current.** [Low] A maturing product has a governance spine: a `SECURITY.md` (model + backlog), owner-gated deploy, "agents propose, the owner disposes" decision rights, and specialist gates (Gus/Lex/Iris). Map these to CSF GV.OC/GV.RM/GV.RR/GV.PO/GV.OV. Verify `SECURITY.md` is up to date and the backlog reflects reality. Realistic target tier = **Tier 2 (Risk-Informed)**, not Tier 3/4. Defensive fix: keep `SECURITY.md` + the backlog as the living risk register; review it each review cycle.
2. **CSF Identify — a current asset + data inventory exists.** [Med] You can't protect what you haven't listed. Confirm a one-page inventory: data assets (buyer PII in your contact/quote/phone tables, payment session ids, telephony numbers), secrets (your env reference doc), subprocessors (item 12), and trust boundaries. Cross-ref the data-and-rls playbook's table. Maps CSF ID.AM-01/02/08, ID.RA-01. Defensive fix: maintain the inventory; re-check after each new integration or table.
3. **CSF Protect is the strong function — confirm it, don't rebuild it.** [Low] Access control (signed cookies, per-route auth, RLS), data-in-transit (HSTS/TLS in your app config), and platform hardening (CSP/X-Frame/nosniff/Referrer-Policy) already satisfy CSF PR.AA-01/05, PR.DS-02, PR.PS-01. Verify they're intact; defer the deep checks to the auth, data-and-rls, and web-hardening playbooks. Defensive fix: none needed beyond keeping them green.
4. **CSF Detect is the realistic gap for a small product — confirm what telemetry exists.** [High] There is no SIEM and that's acceptable at this scale — but "no detection at all" is not. Verify the lightweight detect stack is actually usable: host runtime logs, database logs + advisors, payment-processor dashboard (disputes/fraud), telephony + email dashboards. Maps CSF DE.CM-01, DE.AE-02. Defensive fix: document where each signal lives and what "abnormal" looks like (spike in LLM/email spend → the rate-limit gap; spike in 401s on owner routes; payment dispute surge). Don't buy a SOC; write the one-pager.
5. **CSF Respond + Recover — a 1-page incident-response + backup runbook exists.** [Med] Confirm there's a written answer to "secret leaked / payment key compromised / buyer data exposed / refund-dispute fraud": who decides (the owner), rotate-secret steps (dedicated signing secret → invalidates all cookies), and DB backup/PITR status. Maps CSF RS.MA-01, RC.RP-01. Defensive fix: write a short IR runbook + verify point-in-time-recovery is enabled; this is likely a genuine gap to flag to the owner.
6. **Threat-model the real data flows with STRIDE — every trust boundary has a named control.** [High] Build the DFD once and keep it: (a) buyer → checkout → webhook → signed cookie → owner tools → database; (b) user-supplied URL → SSRF-guarded fetch → external site (scan); (c) buyer/owner → LLM features → provider; (d) cron → broad reads. For each STRIDE category confirm the control: Spoofing→webhook signature construction / telephony validation / signed cookie; Tampering→server-computed prices, field allowlists; Repudiation→logs (item 4); Info-disclosure→RLS + explicit selects; DoS→rate limits (in-memory today, backlog) + SSRF byte cap; Elevation→owner-id scoping. Defensive fix: any DFD edge crossing a trust boundary with no control = a finding; record it, don't exploit it.
7. **MITRE ATT&CK coverage is right-sized to the surface, not a full matrix.** [Med] Map only the techniques that fit a single-host + managed-DB + hosted-payment app and confirm a prevent/detect control for each: T1190 exploit public-facing app (CSP/headers, input validation, framework upgrade clears the audit incl. any image-optimizer DoS); T1078 valid accounts (forged-cookie resistance = HMAC + signing-secret fail-closed in prod); T1199 trusted relationship (webhook signature verification); T1530 data from cloud storage (RLS + service key server-only); T1567/T1041 exfiltration (excessive-data-exposure checks). Defensive fix: treat any technique with no control as a backlog item; don't run adversary-emulation/Atomic-Red-Team tooling against production.
8. **SOC 2 readiness map — know which TSC you'd pass today vs. fail.** [Med] Self-score against the AICPA Common Criteria (no auditor needed yet): CC6 Logical Access = **strong** (signed cookie, RLS, per-route auth, owner-id scoping); CC1/CC2/CC3 Governance = **partial** (SECURITY.md exists, no formal policies); CC7 System Operations = **weak** (item 4 detection gap); CC8 Change Management = **gap** (item 9); CC9 Vendor Risk = **partial** (item 12). Defensive fix: produce the gap table as the readiness artifact; pursue an actual Type II only when a customer contract requires it.
9. **SOC 2 CC8 + CI is the highest-value posture gap — confirm the change pipeline.** [High → CI backlog item] The SOC2 evidence pattern is "every production change had a reviewed/approved PR + tests passed." If you commit per-task on `main` with NO CI today (the "stand up CI" backlog item: tests + typecheck + lint + build + dependency audit + auto-updates), verify the ship bar (tests green, config-validate clean, clean build) is actually run before each deploy, and track CI as the fix. Defensive fix: stand up CI so change-management evidence becomes automatic; until then, log the manual ship-bar run.
10. **PCI DSS scope stays SAQ-A — confirm you never touch raw card data.** [Critical] The payment processor owns nearly all PCI scope because card entry happens on its hosted surface; you compute prices server-side and the client cannot set price/coupon. Verify NO PAN, CVV, or full track data is ever stored, logged, or present in your source, database tables, or host logs (grep for card-like patterns; confirm only the processor's session/customer ids are persisted). Maps PCI Req 3 (no prohibited storage) + the SAQ-A eligibility test. Defensive fix: any raw-card handling would blow scope from SAQ-A to SAQ-D — keep all card UI on the processor's hosted surface; this is a hard line.
11. **PCI Req 4 + 6.4 — transmission + the payment surface are hardened (already mostly done).** [Low] Confirm TLS-only (HSTS in your app config), the webhook authenticity control (signature construction with the webhook secret), and the security headers (CSP/X-Frame/nosniff). Maps PCI Req 4.2.1 (strong crypto in transit) + 6.4.x (protect public web app). Defensive fix: keep headers + signature verification intact; CSP tightening (nonce, drop `unsafe-inline`) is a backlog item.
12. **Vendor / supply-chain risk is inventoried (GV.SC / SOC2 CC9 / PCI Req 12.8).** [Med] Confirm a subprocessor list with what data each touches: host (hosting/logs), database (PII), payment processor (payments), telephony (phone/PII), LLM provider (prompts — never send secrets/PII you don't need), email provider (email). Plus dependency hygiene: dependency audit clean and auto-updates on, and any framework upgrade. Defensive fix: maintain the subprocessor list for DPAs/buyer security questionnaires; resolve audit findings before they pile up.
13. **Privacy / data lifecycle is mapped (LINDDUN + SOC2 Privacy).** [Med] Buyer PII (contact/quote/phone tables) has a collection→retention→deletion path. Verify a privacy policy exists, a data-deletion request can be honored (which tables, scoped by owner id), and LLM features don't over-collect (LINDDUN Linkability/Identifiability/Disclosure). Maps SOC2 Privacy P-series + any multi-tenant per-tenant data boundary. Defensive fix: document retention + a deletion runbook; minimize what's sent to the LLM/third parties.

## Framework tags
- **NIST CSF 2.0:** GV.OC-01, GV.RM-01, GV.RR-01, GV.PO-01, GV.OV-01, GV.SC-06/07 (govern/supply-chain); ID.AM-01/08, ID.RA-01, ID.IM-04 (identify); PR.AA-01/05, PR.DS-02/10, PR.PS-01/04 (protect); DE.CM-01, DE.AE-02/06 (detect); RS.MA-01 (respond); RC.RP-01 (recover). Implementation-Tier target = Tier 2 (Risk-Informed).
- **SOC 2 / AICPA TSC:** Security Common Criteria CC1–CC9 (CC6 access strong, CC7/CC8 are the gaps), plus Confidentiality (C1) and Privacy (P-series) if a buyer contract requires them.
- **PCI DSS 4.0.1:** SAQ-A merchant scope; Req 3 (no prohibited card storage), Req 4 (strong crypto in transit), Req 6.4 (protect public web app), Req 12.8 (subprocessor management). The payment processor owns the CDE.
- **OWASP:** STRIDE + LINDDUN threat modeling; OWASP Top 10 A01 Broken Access Control, A05 Security Misconfiguration (cross-ref the per-domain playbooks for the deep checks).
- **MITRE ATT&CK (Enterprise/Cloud, right-sized):** T1190 (exploit public-facing app), T1078 / T1078.004 (valid/cloud accounts), T1199 (trusted relationship — webhooks), T1530 (data from cloud storage), T1213 (data from info repositories), T1567/T1041 (exfiltration).
- **NIST AI RMF + MITRE ATLAS:** MAP-5.1, MEASURE-2.7, MANAGE-2.4; AML.T0070, AML.T0066, AML.T0082 — apply when threat-modeling the LLM-powered + multi-tenant features (cross-ref the LLM/agent playbook).
- **MITRE F3 (fraud):** the source skills carry no F3 tags; F3 (FA0001 Positioning / FA0002 Monetization, e.g. payment/refund-dispute fraud) is relevant only as a cross-ref to the payment-integrity playbook, not this posture map.

## Source-skill taxonomy (read deeper, never execute)
- `performing-nist-csf-maturity-assessment` — **primary.** Mine the 6-Function/22-Category taxonomy and the 4 Implementation Tiers to self-score at the right altitude (Tier 2) and avoid the "assess tech only, ignore Govern" pitfall. Drives checks 1–5.
- `performing-threat-modeling-with-owasp-threat-dragon` — **primary.** Mine the STRIDE/LINDDUN element-to-threat tables and DFD method; apply to the real data flows (checks 6, 13). Skip the Docker/desktop-tool install — the value is the methodology, not the app.
- `implementing-threat-modeling-with-mitre-attack` — **primary.** Mine the threat-actor→TTP→detection-coverage→gap-analysis loop; right-size it to your handful of relevant techniques (check 7). Do NOT run the Caldera/Atomic-Red-Team emulation steps it lists.
- `performing-soc2-type2-audit-preparation` — **primary.** Mine the CC1–CC9 breakdown, the change-management PR-approval evidence pattern, and continuous-control-drift checks; translate to a readiness gap table (checks 8, 9). Ignore the AWS/Okta/boto3 evidence-collector code — wrong stack.
- `implementing-pci-dss-compliance-controls` — **primary.** Mine the 12 requirements + scoping/CDE concept to confirm SAQ-A eligibility and the no-prohibited-storage rule (checks 10, 11). Ignore the 42-week enterprise rollout plan — if your scope is processor-hosted it stays small.

## Out of scope / do NOT do
- Do not chase certification or build a full GRC/SOC/SIEM program — for a one-owner product the deliverable is a readiness map + gap list, not an audit. Pursue a real SOC 2 Type II or PCI ROC only when a signed customer contract requires it.
- Never run the threat-modeling validation tooling as attacks: no MITRE Caldera, Atomic Red Team, adversary emulation, or `Invoke-AtomicTest` against your app, a sibling product, or any third party. Cyrus is read-only/advisory — extract the control and how to confirm it.
- Never expand PCI scope: do not introduce any flow that stores, logs, or transmits raw PAN/CVV/track data — that converts the assessment from SAQ-A to SAQ-D. Keep all card entry on the processor's hosted surface.
