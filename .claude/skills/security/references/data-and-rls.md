# Database RLS + data-access review (Playbook 3) — security reference

> Defensive reference for Cyrus (security). Condensed from a public defensive-security skills library (Apache-2.0), grounded in a typical modern web stack. DEFENSIVE ONLY: checklists, detections, and hardening to VERIFY on systems you own. Never run offensive/exploit steps; for any 'exploiting-*'/'testing-*' source, extract only the defensive control + how to confirm it, not an attack recipe.

## What Cyrus checks here
How the app reads and writes data: Row Level Security (RLS) posture, the service/admin database client that bypasses RLS, owner-scoped query isolation in the privileged tools, and API responses that may over-return database columns (excessive data exposure / BOPLA, OWASP API3:2023). It owns confirming RLS is actually ON for anon-reachable tables and owns the "full server-only split for the admin client" backlog item. It also covers per-tenant isolation in any multi-tenant product (a DB-enforced membership check + an active-tenant cookie). Surfaces touched: your database client module, your access/entitlement module, every API route that uses the admin client, and the database project itself. *(Examples below assume a Postgres-with-RLS database such as Supabase; translate to your DB's equivalent access-control model.)*

## Checklist
1. **RLS is ON for every anon-reachable table.** [Critical] The browser uses the public anon key, which is subject to RLS — but RLS that was never enabled is a silent open door. Confirm via your DB's security lint / dashboard that EVERY table has RLS enabled and a deny-by-default policy, especially any table holding PII, entitlements, or access-revocation state. Defensive fix: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY;` + an explicit policy (or no policy = deny-all) on each.
2. **Service/admin key only ever runs server-side.** [Critical → backlog: server-only split] The admin client bypasses RLS; if it relies on a *runtime* `typeof window` guard, verify the key is NOT a public-prefixed env var (so it is never inlined into the client bundle) and that no `'use client'` component imports the admin client or the access module. Defensive fix: add a `server-only` import to those modules so a client import becomes a build-time error, not a runtime hope.
3. **Every owner-tool query is scoped by the owner's id.** [Critical] If there is no route middleware, each route self-enforces. Confirm every admin `.from(...)` read/update/delete in the owner routes chains `.eq("<owner_key>", s)` where `s` is the owner id derived **from the verified cookie**. Pattern-match new routes against a known-good one. Defensive fix: any owner query missing the owner-scope filter is a BOLA hole — add the scope or justify it as a validated chained lookup (see #4).
4. **Chained ("trust the parent check") queries must be unreachable without the ownership check.** [High] A child-table read scoped only by a parent id (not the owner id) is safe ONLY if an earlier line already verified the parent row is owner-scoped. Verify the ownership check always precedes and gates the child query (no early-return skip, no reordering). Defensive fix: prefer also scoping the child by the owner id, or assert the parent row was found before any child fetch.
5. **Write paths use an explicit field allowlist (no mass assignment).** [High — BOPLA write side] The canonical correct pattern builds the patch object from only the specific writable fields (length-capped) and never spreads `body` into `update()`. Verify NO route does `admin.from(t).update(body)` / `.insert(body)` / `{...body}` with raw client JSON — that would let a caller inject the owner key, stage, status, amount, etc. Defensive fix: per-route writable-field allowlist; never bind the request body directly to the row.
6. **Prices, statuses, and entitlement fields are server-set, never client-set.** [Critical] Payment amounts are computed server-side and the client cannot set price/coupon. Confirm no API accepts `amount`, `status`, `tier`, `subscription_tier`, `is_verified`, or `role`-like fields from the request body and persists them. Defensive fix: derive all money/entitlement fields from server/source-of-truth state; reject or ignore them in input.
7. **Entitlement gates fail OPEN today — confirm the blast radius and track the fix.** [High → backlog: fail-closed gates] Gates that `catch {}` → return access on a DB error (so a DB blip never locks out a paying user) must not be *forceable* (e.g. by inducing DB errors) to bypass access-revocation lockout. Defensive fix: fail closed (or grace-window) for money-spending/PII gates and reconcile from the source of truth; keep read-only/content gates fail-open if desired.
8. **API responses don't leak internal columns (excessive data exposure).** [Med — OWASP API3] If owner tools `select("*")` and return the row to the client, verify those tables hold no secret/internal columns (raw tokens, other tenants' data, internal scores, provider auth, webhook payloads) that get shipped to the browser. Defensive fix: select explicit columns or a view/DTO for client-facing responses; keep `select("*")` only for server-internal use.
9. **Cross-owner / unscoped scans run only in trusted server contexts.** [High] Cron and webhook paths intentionally read broadly (e.g. `.limit(200)` across all rows). Confirm these are reachable ONLY via a verified cron Bearer secret (constant-time) or a verified payment/telephony signature — never from a user cookie. Defensive fix: assert the cron/webhook auth at the top of every broad-scan route; never expose an unscoped read to a user/owner route.
10. **The cookie-signing secret (the trust anchor) fails closed in prod.** [Critical] If all access/owner cookies are HMAC-signed with a secret resolved from a fallback chain (else throws in production; dev-only fallback), verify production has a real secret set so the guessable dev fallback is never used, and that the dev fallback string never appears in a prod bundle. Defensive fix: set a dedicated, rotatable signing secret and confirm the prod throw path.
11. **Tokens have no expiry — confirm the compensating revocation works.** [Med → backlog: token exp] If access/owner cookies carry `iat` but no `exp`, the only kill-switch is the access-revocation table / subscription-status lookup. Verify those tables + the gate lookups are intact and that a refund/dispute/cancel actually revokes. Defensive fix: add `exp` + a refresh path; for now confirm revocation rows are written on the payment refund/dispute/cancel webhooks.
12. **Multi-tenant per-tenant isolation is enforced in the DB, not just the app.** [Critical] A multi-tenant product should use per-tenant RLS via a membership function (e.g. SECURITY DEFINER) and a forge-proof active-tenant cookie, covered by an isolation test. Verify: the SECURITY DEFINER function has a pinned `search_path` (cannot be hijacked), RLS is enabled on every tenant table, the active-tenant cookie is signed/verified server-side, and the isolation test passes (a tenant cannot read another tenant's rows). Defensive fix: keep the isolation test in the ship bar; treat any cross-tenant read as Critical.
13. **DB posture lint is clean (the small-product CSPM check).** [Med] For a single host + managed database (not multi-cloud), translate CSPM into: run your DB's security + performance advisors / dashboard linter; confirm no "RLS disabled", no "policy allows public", no "SECURITY DEFINER without search_path", no exposed database port, no overly-broad anon grants. Confirm only public-prefixed env reaches the browser and nothing secret is committed (env-only). Defensive fix: resolve every advisor; re-check after each migration.

## Framework tags
- **OWASP API Security Top 10:** API3:2023 Broken Object Property Level Authorization (excessive data exposure + mass assignment), API1:2023 BOLA (owner-scope isolation), API6:2023 unrestricted access to sensitive business flows.
- **OWASP Top 10:** A01 Broken Access Control, A05 Security Misconfiguration.
- **OWASP LLM/Agentic:** relevant for any multi-tenant MCP/tenant data access (excessive agency / sensitive-info disclosure) — cross-ref the LLM/agent playbook.
- **NIST CSF 2.0:** PR.PS-01, PR.DS-10, ID.RA-01, DE.CM-01 (data exposure/BOPLA); PR.AA-01, PR.AA-05, PR.IR-01, GV.PO-01 (zero-trust access); PR.IR-01, ID.AM-08, GV.SC-06 (posture/CSPM).
- **MITRE ATT&CK:** T1190 (exploit public-facing app), T1213 (data from info repositories), T1212, T1552.001 (creds in files), T1078 / T1078.004 (valid/cloud accounts), T1530 (data from cloud storage), T1537, T1580.

## Out of scope / do NOT do
- Never run the offensive siblings as attacks: do not execute the BOPLA/excessive-exposure scanner scripts, mass-assignment injection loops, GraphQL introspection probes, or any request-tampering against your app, a sibling product, or third parties. Cyrus is read-only/advisory — extract the control and how to confirm it, never the attack.
- No multi-cloud CSPM tooling (Prowler/ScoutSuite/Security Hub/Defender) and no enterprise CASB/SSPM/Entra conditional-access stack — overkill for a small single-host + managed-DB product; use your DB's advisors + host env hygiene instead.
- Do not weaken the deliberate fail-open behavior or disable RLS to "make a query work"; route any entitlement change through the fail-closed-gates backlog item and owner approval.
