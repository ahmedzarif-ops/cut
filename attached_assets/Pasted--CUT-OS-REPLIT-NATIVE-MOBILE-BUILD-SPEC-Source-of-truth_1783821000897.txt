# 0. READ THIS FIRST

This is a **native mobile application**.

It is not:
- a landing page
- a static HTML prototype
- a responsive website pretending to be an app
- a generic fitness dashboard
- a MyFitnessPal clone
- a design concept with dead controls

Create the project as a **Mobile app artifact** in Replit.

Use the React Native + Expo mobile workflow supported by Replit Agent.

The application must be operable in:
1. Replit iOS Simulator
2. Expo Go on Zarif's real iPhone

The cloud server and PostgreSQL database are the system of record.

Do **not** use `localStorage` as the application database.

Do not choose a web-only framework for the primary client.

# 2. NON-NEGOTIABLE QA RULE

A button is not considered implemented because:
- a handler exists
- TypeScript compiles
- the screen renders
- a component test passes
- Agent says it should work

A primary control is considered implemented only after the actual interaction has been exercised in the **running iOS Simulator**.

Hardware-dependent behavior must also be exercised in **Expo Go on a real phone** when practical.

Do not declare the app ready while any core user flow is dead, silently fails, loses data after reload, or navigates to an empty screen.

Replit automated App Testing for web apps is not a substitute for native-mobile interaction testing.

At the end of the build, create `QA_REPORT.md` containing:
- every acceptance scenario
- PASS / FAIL / BLOCKED
- device or environment used
- exact issue found
- exact fix made
- remaining limitation

Do not write "all buttons work" unless the mandatory mobile acceptance matrix in this spec was actually executed.

# 5. TECHNICAL ARCHITECTURE

## 5.1 Client

Use:
- React Native
- Expo
- TypeScript

Prefer:
- Expo Router for route and tab navigation
- TanStack Query for server state and cache
- a very small local UI-state layer only where necessary
- React Hook Form + Zod or equivalent typed form validation
- Expo Haptics for restrained native feedback
- Expo Notifications if reliable in the current mobile artifact and real-device preview
- Expo SecureStore for client-side credential/token material when required by the selected auth path
- React Native Safe Area Context
- a chart library compatible with Expo Go, or React Native SVG-based charts

Avoid:
- a large Redux store duplicating server data
- storing derived macro totals as independent editable state
- multiple state-management libraries
- unsupported native packages that require a custom dev build for MVP unless there is no reasonable Expo-compatible alternative

## 5.2 Server

Use a TypeScript server supported in the Replit project.

Responsibilities:
- authenticated API routes
- user ownership validation
- database reads/writes
- deterministic next-action calculation
- deterministic weekly review calculation
- PR calculations
- reminder preference persistence
- data export
- validation
- error normalization

Do not trust `user_id` sent from the client.

Resolve authenticated user identity server-side.

## 5.3 Database

Use Replit Database / PostgreSQL.

Use migrations.

Use a typed ORM that is compatible with the Replit scaffold. Drizzle is preferred if it fits the generated project cleanly.

The database is the source of truth for:
- profile
- weight entries
- measurements
- meal templates
- food entries
- workout sessions
- workout sets
- daily logs
- weekly reviews
- reminder preferences

Derived values should be calculated from source entries.

## 5.4 Authentication decision gate

Start with Replit Auth through Agent because this application is initially for Zarif and he already has a Replit account.

Before building the full authenticated data layer:

1. implement authentication
2. launch the native app in the iOS Simulator
3. verify sign-in can complete in the native workflow
4. verify the server receives a stable authenticated identity
5. verify a database row can be created and read back for that identity
6. verify user A cannot access another user ID by changing a client request

If Replit Auth is not reliable or supported for the running native artifact, use another Replit-supported managed auth option such as Clerk.

Do not hand-roll password authentication.

Record the final auth decision and reason in `ARCHITECTURE.md`.

## 5.5 Client/server boundary

The client may:
- optimistically update UI
- temporarily queue an entry
- cache recent reads
- animate completion

The server owns:
- authorization
- canonical values
- duplicate protection
- database writes
- weekly review logic
- next-action rules
- PR persistence
- source-of-truth calculations

## 5.6 Offline and poor-connection behavior

Fitness logging must not feel dangerous in a gym with weak service.

For create/update operations:
- preserve form data until confirmed saved
- show pending state
- retry safe transient failures
- use an idempotency key for create mutations
- do not create duplicate food entries or workout sets on retry
- roll back optimistic totals on permanent failure
- show a visible "Not synced" state instead of silently pretending a failed mutation succeeded

A full offline-first CRDT architecture is not required.

```ts

  type:
    | "WEIGH_IN"
    | "MEAL_ONE"
    | "TRAIN"

    | "STEPS_GAP"

  title: string;
  subtitle: string;
  ctaLabel: string;
  route: string;
  payload?: Record<string, unknown>;
  priority: number;
};
```

Do not generate next actions with an LLM.

## 9.3 Suggested priority rules

Evaluate the authenticated user's local date and timezone.

Default timezone:
`America/Chicago`

6. step target materially behind later in the day
7. closeout due in the evening
8. day complete

Default Meal 1 guidance window:
approximately 12:00 PM–3:00 PM

after 8:00 PM

## 13.5 Adaptive cut coach

This is deterministic.

No LLM.

### Target lane

If weekly average loss is between 1.5 and 2.2 lb:

Recommendation:
**Hold course**

Reason:
Rate of loss is in the target lane.

Do not recommend more cardio or fewer calories.

### Slow

If weekly average loss is less than 1.0 lb:

Do not change the plan after one weekly window.

If this occurs for **two consecutive full weekly comparisons**:

Recommend one:
- reduce calorie target from 1,800 to 1,650
- add 2,000 average daily steps

Selection rule:
- if calorie logging compliance is poor, recommend `Tighten food logging` before changing targets
- if average calories are materially above target, recommend `Hit the existing calorie target`
- if calories are on target but steps are consistently below target, recommend `Add 2,000 daily steps`
- if calories and steps are compliant but loss remains slow, recommend `Reduce calorie target to 1,650`

Exactly one primary recommendation.

### Too fast

If weekly average loss exceeds 2.5 lb after the first 14 days:

Recommendation:
**Add 100–150 calories**

Reason:
Protect training performance and lean mass.

If only one abnormal week occurs:
watch the next window before escalating further.

# 20. CALCULATION RULES

## 20.1 Daily food totals

Derive from food_entries for user/date:

- calories = sum calories
- protein = sum protein_g
- carbs = sum carbs_g
- fat = sum fat_g
- fiber = sum fiber_g

## 20.2 Macro action completion

Default rule for the daily `Macro target` action:

Complete when:
- calories >= 1,450
- calories <= calorie target + 100
- protein >= 150g

These thresholds are Zarif-specific defaults.

Keep target values profile-driven where possible.

Do not mark the macro action complete solely because calories are low.

## 20.3 Movement completion

Complete when:
- planned workout/session is completed

OR, on non-strength recovery/cardio days:
- planned movement is marked complete

OR:
- steps >= steps target when there is no incomplete required strength session

Do not allow 10,000 steps to silently replace a scheduled Monday strength session.

## 20.4 Meal 1 completion

Complete when a food entry for the date has:
- meal_type `MEAL_ONE`

Do not infer Meal 1 from time alone.

## 20.5 Daily status score

Score 0–5 from:
- weight logged
- Meal 1 logged
- macro target met
- movement requirement met
- closeout done

## 20.6 Streak

Count consecutive local calendar days ending today or yesterday where:
daily status score >= 4

If today is still in progress and below 4:
do not prematurely break yesterday's streak.

## 20.7 Estimated 1RM

Use Epley:

For reps == 1:
`e1rm = weight`

For reps > 1:
`e1rm = weight * (1 + reps / 30)`

Round for display.
Preserve reasonable precision internally.

## 20.8 Weight trend

7-day moving average:
mean of available daily weight entries in the rolling 7-day period.

For formal weekly review comparisons:
require sufficient coverage.

Preferred:
at least 5 weigh-ins in each 7-day window.

If coverage is lower:
label confidence as incomplete and do not make aggressive target changes.

# 21. ERROR HANDLING

Every mutation failure must:

1. keep entered form values
2. show a clear user-facing error
3. distinguish retryable vs validation errors where possible
4. allow retry
5. avoid duplicate creation

Never:
- silently dismiss a modal on failed save
- show success before a permanent failure is known
- leave totals showing optimistic data after rollback
- lose a logged workout set because the keyboard closed

Use a consistent native toast/banner pattern.

Initial authenticated loads:
- use skeletons for major surfaces

Small mutations:
- keep screen context visible
- disable only the relevant save action
- show a compact progress state

- adding water
- logging a set
- editing food
- logging weight

Every empty state should point to the next useful action.
